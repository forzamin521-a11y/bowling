# Design Review: 경기도볼링협회 SaaS 전체 사용자/관리자 흐름

**Reviewed:** 2026-06-29
**Focus:** 사용자 흐름, 버튼 위치, 뒤로가기 흐름, 불편 지점, 일관성
**Platform:** Responsive (모바일 우선 공개 페이지 + 데스크톱 위주 관리자)

## Summary

전반적인 시각 디자인 완성도는 높습니다(일관된 카드/배지/그라데이션, 다크모드, 호버 마이크로인터랙션). 다만 **기획-구현 불일치 1건(예정 대회 클릭 차단 누락)**, **뒤로가기 흐름의 단절(딥링크 진입 시)**, **관리자 작업 단계 안내 부재**가 실제 사용성을 떨어뜨립니다.

---

## Critical

### C1. 예정(upcoming) 대회가 클릭되어 빈 페이지로 진입 — 기획 불일치
- **위치:** `app/page.tsx:73-118`, `app/tournaments/[id]/page.tsx`
- **문제:** 기획서(개발서.md 12-15행)는 "예정이라면 클릭되지 않고 '준비중' 안내만 뜬다"고 명시. 그러나 현재 모든 대회 카드가 `<Link>`라 예정 대회도 클릭되어 종별/순위(빈 데이터) 페이지로 진입함.
- **영향:** 사용자가 아직 열리지 않은 대회에서 빈 순위표를 보고 "고장났나?"로 오해.
- **권장:** upcoming 상태일 때 카드를 비활성(`<div>` + `cursor-not-allowed` + "준비중" 표시)으로 렌더. 또는 상세 페이지에서 upcoming이면 종별 대신 안내 카드 노출.

## Major

### M1. 딥링크/새로고침 시 뒤로가기 경로의 신뢰성
- **위치:** 공개 페이지의 상단 `<Link>` 뒤로가기 (`tournaments/[id]/page.tsx:71`, `[categoryId]/page.tsx:65`, `[eventId]/page.tsx:471` 등)
- **문제:** 뒤로가기를 브라우저 history가 아닌 명시적 상위 링크로 처리한 것은 좋음(딥링크 안전). 다만 단계가 4단(목록→대회→종별→세부종목→순위)으로 깊은데 **breadcrumb이 없어** 현재 위치 파악이 어렵고, 한 단계씩만 거슬러 올라갈 수 있음(목록에서 3번 클릭 필요).
- **권장:** 공개 상세 페이지 상단에 간단한 breadcrumb(`대회목록 / 대회명 / 종별 / 세부종목`) 도입. 최소한 "대회 목록"으로 한 번에 가는 링크 병기.

### M2. 관리자 작업 단계 안내(progress) 부재
- **위치:** `app/admin/(authenticated)/tournaments/[id]/page.tsx:181` (텍스트 한 줄로만 안내)
- **문제:** 선수등록 → 팀편성/레인배정 → 점수입력 순서가 작은 텍스트로만 안내됨. 단계 완료 여부(체크) 표시 없음.
- **권장:** 대회 상세에 스텝 인디케이터(`✓ 선수등록 → □ 레인배정 → □ 점수입력`)와 "다음 단계" primary 버튼.

### M3. 깊은 관리자 페이지(점수입력 등)의 탈출/이동 경로
- **위치:** `app/admin/.../events/[eventId]/scores/page.tsx`, `.../lanes/page.tsx:90-117`
- **문제:** 상단에 4개 버튼이 flex-wrap. 모바일에서 줄바꿈되어 primary(점수입력 저장)와 secondary(순위 보기) 구분이 흐림. breadcrumb 없음.
- **권장:** breadcrumb 추가 + primary 액션 1개만 강조, 나머지는 보조 스타일/드롭다운.

### M4. 삭제 등 위험 액션의 일관성
- **위치:** `delete-button.tsx`, `events-section.tsx:204`, `players-table.tsx:185`
- **문제:** 삭제 UI가 곳마다 다름(텍스트 버튼 vs 아이콘만), 일부는 브라우저 `confirm()` 사용 → 디자인 시스템과 불일치.
- **권장:** 공통 `ConfirmDialog`로 통일, destructive 스타일 일관 적용.

### M5. 에러 피드백 위치 불일관
- **위치:** `login-form.tsx:76`(인라인) vs `registration-form.tsx`(toast)
- **권장:** 폼 검증 에러는 인라인 Alert, 서버 작업 결과는 toast로 역할 분리.

## Minor

- **m1. 빈 상태 단조로움:** 공개/관리자 빈 상태가 텍스트뿐. 아이콘+CTA를 가진 `EmptyState` 컴포넌트화. (`app/page.tsx:60`, admin 다수)
- **m2. 등록 버튼 텍스트:** `registration-form.tsx:237` `${filledNames.length || ""} 등록` → 0명일 때 " 등록"으로 어색. `N명 등록`/비활성 처리 권장.
- **m3. 사이드바 용어:** `sidebar.tsx:27` "선수 마스터" → "선수 관리/전체 선수"가 직관적.
- **m4. 모바일 테이블:** `players-table.tsx`, 점수/레인 보드가 가로스크롤·드래그앤드롭 → 모바일 카드형/대체 입력 모드 고려.
- **m5. 헤더 구조 표준화:** 관리자 상세 페이지마다 헤더 구성(뒤로가기/제목/설명/액션) 제각각 → `PageHeader` 컴포넌트화.

## Positive

- 상태 우선 정렬(진행중→예정→종료) + 진행중 ping 애니메이션 배지: 정보 우선순위 우수 (`page.tsx:14`, `status-badge.tsx`).
- 뒤로가기를 history가 아닌 명시 링크로 → 딥링크/새 탭에서도 안전.
- 다크모드, 모바일 드로어 자동 닫힘, skip-link 등 기본기 충실.
- 읽기/쓰기 최적화를 위한 게임 마감 기반 랭킹 캐시 반영 — 기획 의도 정확히 구현.

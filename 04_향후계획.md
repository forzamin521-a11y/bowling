### 2-2. 미실행 마이그레이션 적용

Supabase SQL Editor에 아래 SQL 붙여넣고 Run:

```sql
-- 0003_move_lanes_to_events.sql 내용
alter table tournament_events
  add column if not exists lane_start smallint,
  add column if not exists lane_end   smallint;

do $$
begin
  if not exists (
    select 1 from pg_constraint
    where conname = 'tournament_events_lane_range_check'
  ) then
    alter table tournament_events
      add constraint tournament_events_lane_range_check
      check (
        (lane_start is null and lane_end is null)
        or (
          lane_start is not null
          and lane_end is not null
          and lane_start >= 1
          and lane_end   >= lane_start
        )
      );
  end if;
end $$;

drop view if exists tournaments_with_status;

alter table tournaments
  drop column if exists lane_start,
  drop column if exists lane_end;

create or replace view tournaments_with_status as
select
  t.*,
  coalesce(
    t.status_override::text,
    case
      when current_date < t.start_date then 'upcoming'
      when current_date > t.end_date   then 'finished'
      else 'ongoing'
    end
  ) as status
from tournaments t;
```

### 구현 시 결정한 것 (기획 대비)

- 마스터 매칭은 (이름+시군+소속명) **정확 일치 자동 매칭**으로 단순화 — 모호 케이스(이름만 같고 소속 다름)는 자연히 신규 생성되므로 별도 확인 모달 생략
- 폼은 코드베이스 기존 스타일(plain state + server action zod)로 작성 (RHF 미사용)
- `database.types.ts`의 `tournament_players` Update 타입에 `player_id/region_id/affiliation_name` 추가

### (원래 계획) 데이터 흐름

### 데이터 흐름

```
선수등록 화면
  1) 시/군 드롭다운 선택 (regions 31개)
  2) 소속 입력 (자유 입력 + 자동완성)

     - tournament_players에 추가
     - registered_order = 현재 시점 마지막 + 1
```

- 중앙: 등록된 선수 표 (선수번호, 시/군, 소속, 팀라벨, 이름, 수정, 삭제)
- 우측 또는 모달: 신규 등록 폼 (위 흐름)

### 구현 단위 작업

2. `app/admin/(authenticated)/tournaments/[id]/players/page.tsx` — 메인 화면
3. `.../players/registration-form.tsx` — 다인원 등록 폼 (RHF + zod)
4. `.../players/players-table.tsx` — 검색/필터/수정/삭제

### 신경 쓸 부분

- **선수번호 재사용 금지**: 삭제 시 빈 번호 그대로 둠. `next_player_number()` DB 함수가 이미 그렇게 동작.
- **팀라벨 자동 재계산**: 트리거 `after_tp_change`가 동작 중. 등록/삭제 후 표 새로고침 필요.
- **마스터 선수 매칭 UX**: 동일 (시군+소속+이름)이면 자동 매칭, 다르면 새 player 생성. 모호하면 사용자 확인.
- **검색 인덱스**: `pg_trgm` GIN 인덱스 활용 — `name ilike '%검색어%'` 또는 `name % '검색어'`.
- **검증**: 대회 상태가 `finished`면 등록/수정/삭제 막을지 결정 (기획안에 명시 안 됨). 일단 허용.

### 구현 시 결정/주의

- **DB 제약**: `tournament_team_members.unique(tournament_player_id)` → 한 선수는 대회 전체에서 **한 팀에만** 편성됨. 따라서 미편성 후보 = 아직 어떤 팀에도 없는 선수 (이벤트 무관 전역)
- **카테고리 필터 없음**: `tournament_players`에 종별 정보가 없어, 후보 목록을 종별(연령/성별)로 거르지 않음. 관리자가 같은 그룹 내에서 직접 선택 (기획안과 동일)
- `team-by-numbers.tsx`는 별도 파일 대신 `team-builder.tsx` 안 탭으로 통합
- `setLineup`(후반전 교체)은 Phase 7 점수 화면에서 구현 예정. 여기선 편성 시 5인조 초기 라인업(모든 게임 동일)만 생성
- 5인조 5명 팀도 `event_lineups` 생성(전원 starter) — 점수 합산 로직 일관성 위해

### 동작 확인 (5분)

1. 세부종목 행 "레인 배정" → 사용 레인 미설정이면 안내, 설정됐으면 보드 노출
2. 미배정 풀에 팀 칩(원자) + 개인 칩 노출
3. **랜덤 배정**: 모든 레인 인원 균등(편차 ≤ 1 지향), 인원 초과 시 경고 토스트
4. **드래그**: 팀/개인을 레인으로 끌어다 놓기, 풀로 되돌리기. 한 레인 6명 초과 드롭은 차단
5. 레인 헤더 `n/6` 배지, 초과 시 빨강 + 상단 경고 배너
6. **저장** → 새로고침해도 유지, **전체 초기화** → 저장본까지 삭제
7. 미리보기 표: 이동칸>0이면 1G~NG 각 게임 레인 번호(래핑), 0이면 "이동 없음"

### 구현 시 결정/주의

- **레인 풀 = 이 세부종목 팀(원자) + 이 종목 팀에 속하지 않은 개인 선수.** 개인전은 팀이 없으니 전원 개인. (스키마에 종별-선수 링크가 없어 개인은 종목 무관 노출 — 관리자가 해당 종목 인원만 배치)
- `team-by-numbers`/수동 분리 파일 대신 `lane-board` 하나로 통합
- 저장은 이벤트의 lane_assignments 전체 삭제 후 재삽입 (팀=멤버 전원, 개인=레인별 NULL 그룹)
- 하드캡(6)은 드롭 차단 + 랜덤 경고. 저장 자체는 막지 않음(스키마 제약 없음)
- `max_per_lane`은 `tournaments.settings.max_per_lane` (기본 6)

### 화면

- 좌측: 미배정 팀/선수 리스트
- 중앙: 사용 레인 카드 (세부종목의 `lane_start ~ lane_end`)
- 상단: [랜덤 배정] [전체 초기화]

### 랜덤 알고리즘 (기획안 그대로)

1. 사용레인 수 L = `lane_end - lane_start + 1`
2. 팀 단위 셔플 → 한 팀은 같은 레인
3. 미편성 선수 셔플 → 남은 자리 분산
4. 모든 레인 인원 편차 ≤ 1
5. 하드 캡: 한 레인 최대 6명 (`settings.max_per_lane`, 기본 6)
6. 초과 시 경고

### 드래그앤드롭

- `@dnd-kit/core` + `@dnd-kit/sortable`
- 팀 카드 드래그 → 멤버 같이 이동
- 미편성 선수 개별 드래그
- 같은 팀 쪼개기 시도 시 드롭 차단

## 6. Phase 7 — 점수 입력 + 게임 마감 + 5인조 교체 ✅ 코드 완료 (사용자 동작 확인 필요)

### 구현된 파일

- `.../events/[eventId]/scores/page.tsx` — 레인 배정 기반 채점 대상 로드 (서버)
- `.../scores/score-board.tsx` — 게임탭 + 레인별/전체 토글 + 디바운스 셀 + 마감 컨트롤 + 교체 트리거

- `.../scores/actions.ts` — `upsertScore`, `lockGameAction`/`unlockGameAction`(rpc), `substitutePlayer`
- 세부종목 행에 **점수 입력** 링크

### 동작 확인 (5분)

1. 레인 배정이 없으면 "먼저 레인 배정" 안내
2. 게임 탭(1G…NG) 선택, 레인별(기본)/전체 토글
3. 점수 셀 입력 → 1초 후 자동 저장(테두리 색: 저장중 amber → 저장됨 green), 0~300 외엔 에러
4. [게임 마감] → 확인 → `lock_game` rpc → 랭킹 갱신, 셀 잠김. (super_admin은 [마감 해제] 노출)
5. 5인조 후반 게임 탭에서 "후반전 선수 교체" 카드 → 6명 팀 [교체] → 출전 5명 선택 → 변경

### 구현 시 결정/주의

- **채점 대상 = 레인에 배정된 선수**(lane_assignment_players). 가장 자연스럽고 레인별 입력 UX와 일치
- `refresh_rankings`는 **마감된 게임만** 합산. 팀전 개인랭킹은 팀 멤버만 집계(개인전만 전체) — DB 함수 규칙 그대로
- 셀 저장은 잠금 게임 차단(서버) + 클라이언트 비활성. `revalidate` 미호출(타이핑 중 리로드 방지), 마감/교체 시에만 revalidate
- **5인조 교체 시 레인 이동 생략**: 본 모델은 6명 로스터가 같은 레인에 함께 배정되므로(레인=팀 단위) 물리적 레인 변경 불필요. 라인업 role만 후반 게임 일괄 갱신
- `score-grid`/`score-all`/`lock-button`은 별도 파일 대신 `score-board` 내부로 통합
- `updated_by`는 일단 null(저장 지연 최소화). 필요 시 보강
- 실시간: 관리자 화면은 폴링 미적용(입력 주체가 관리자). 공개 페이지(Phase 8)에서 Realtime 적용 예정

### (원래 계획) UX

### UX

- 진행 중 게임 탭: `1G | 2G | 3G | ...`
- 기본: 레인별 입력 (한 레인 = 한 카드)
- 토글: 전체 한번에 입력
- 입력: 정수 0~300, blur/디바운스 1초 저장
- 게임 헤더 상태 배지: [입력중] / [마감됨]
- 입력 완료 후 [게임 마감] 버튼 → DB 함수 `lock_game(event_id, game)` 호출
  - `game_states` UPSERT
  - `refresh_rankings(event_id)` 실행
- 마감된 게임 셀은 잠금. super_admin만 `unlock_game` 가능

### 5인조 선수 교체

- 후반전 첫 게임 직전 [선수교체] 버튼 노출
- 모달: 6명 로스터 카드, 현재 starter 5명 체크, 변경 가능
- 확정 시 `event_lineups` 후반전 게임들 일괄 UPDATE
- 교체out된 선수는 교체in된 선수의 원래 레인으로 자동 이동 (lane_assignment_players 조정)

### 구현 단위

1. `.../scores/page.tsx`
2. `.../scores/score-grid.tsx` (레인별)
3. `.../scores/score-all.tsx` (전체)
4. `.../scores/lock-button.tsx`
5. `.../scores/substitution-modal.tsx`
6. `.../scores/actions.ts` — `upsertScore`, `lockGameAction`, `substitutePlayer`

### 신경 쓸 부분

- **셀별 UPSERT 충돌 방지**: 단일 셀이라 `(event_id, player_id, game_number)` unique로 충분

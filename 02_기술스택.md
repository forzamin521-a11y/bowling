## 5. Supabase 클라이언트 사용 패턴

| 환경 | 키 | 용도 |
|---|---|---|
| 브라우저(공개 페이지) | `anon key` | `rankings` / `team_rankings` / `lane_assignments_view` 읽기 전용 |
| 브라우저(관리자) | `anon key` + 세션 쿠키 | 관리자 SELECT/INSERT/UPDATE (RLS로 권한 검증) |
| 서버(서버 액션) | `anon key` + cookies | 모든 mutation은 가능한 한 서버 액션에서 수행 |
| 서버(스크립트/특권) | `service_role key` | 시드, 마이그레이션, 게임 마감 트랜잭션(권한 정책에 따라) |

**원칙:**
- `service_role`은 절대 클라이언트에 노출 금지. Next.js 서버 컴포넌트/서버 액션에서만 사용.
- 점수 마감 같은 트랜잭션은 **Postgres 함수**로 캡슐화하고, 서버 액션에서 RPC 호출.

## 6. 인증 & 라우팅 가드

- 미들웨어(`middleware.ts`):
  - `/admin/*` 진입 시 Supabase 세션 확인 → 없으면 `/admin/login`
- 관리자 레이아웃에서 `profiles.role IN ('admin','super_admin')` 추가 검증
- RLS는 DB에서 강제 (UI 우회 차단)

## 7. 실시간 갱신 전략

| 화면 | 방식 | 빈도 |
|---|---|---|
| 사용자 랭킹 | Supabase Realtime 구독 `rankings WHERE event_id = ?` | 게임 마감 시 1회 |
| 사용자 배정레인 | 폴링 30s 또는 Realtime `lane_assignments` | 배정 변경 시 |
| 관리자 점수 입력 | 본인 mutation만 옵티미스틱 업데이트, 외부 변경은 무시 | 즉시 |
| 관리자 대시보드 | 폴링 60s | 충분 |

## 8. 도메인 로직 위치 (DB ↔ App)

| 룰 | 위치 | 이유 |
|---|---|---|
| 팀라벨 재계산 | DB 트리거 + 서버 액션 | 선수 등록/삭제 시 일관성 보장 |
| 선수번호 부여 | DB 시퀀스(대회 스코프) | 동시성 안전 |
| 레인 이동 규칙 (게임별 레인 매핑) | App(`lib/domain/lane-rotation.ts`) + DB 뷰 | 미리보기/점수입력 화면에서 동일 함수 재사용 |
| 랭킹 계산 (개인/팀/5인조) | DB 함수 `refresh_rankings(event_id)` | 트랜잭션 일관성, 성능 |
| 동점자 처리 | DB ORDER BY | 정렬 일관성 |
| 5인조 lineup | DB(`event_lineups` 테이블) + App 검증 | 교체 이력 추적 |

## 12. 성능 / 비용 안전장치

- **읽기 부담:** 사용자 화면은 캐시 테이블만 보고, 인덱스 `(event_id, rank)`로 SELECT 최소화
- **쓰기 부담:** 점수 입력은 셀별 UPSERT but 마감 전까지 랭킹 재계산 안 함
- **Vercel 함수 호출 최소화:** 정적 데이터는 ISR, 동적 데이터는 클라이언트 fetch(Supabase 직접)
- **Realtime 연결 수 제한:** 사용자 페이지는 한 화면당 1개 채널만 구독(`rankings:event_id=...`)

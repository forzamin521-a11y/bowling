# 참가종별 안전한 활성화 관리 설계

## 목표

관리자가 참가종별을 실수로 해제해도 선수, 세부종목, 레인 배정, 점수, 랭킹 데이터가 삭제되지 않도록 하고, 종별을 다시 활성화하면 기존 운영 데이터를 복구한다.

## 현재 문제

관리자 참가종별 UI는 체크박스 변경 즉시 `toggleCategory` 서버 액션을 호출한다. 해제 경로는 `tournament_categories` 행을 직접 삭제하며, 하위 테이블이 `ON DELETE CASCADE`로 연결되어 있어 운영 데이터가 연쇄 삭제될 수 있다. 확인 모달이나 영향 요약도 없다.

## 결정 사항

1. 체크 해제는 삭제가 아니라 `사용 중지`로 처리한다.
2. `tournament_categories.is_active` 상태를 추가하고 기존 데이터는 활성 상태로 백필한다.
3. 사용 중지된 종별은 공개 종별 목록과 기본 운영 선택지에서 숨기되, 관리자 대회 상세에서는 비활성 상태와 복구 동작을 보여준다.
4. 사용 중지된 종별의 선수·세부종목·레인·점수·랭킹은 보존한다.
5. 영구 삭제는 이번 변경에서 일반 토글과 분리한다. 일반 관리자 화면에는 카테고리 삭제 액션을 제공하지 않으며, 향후 영구 삭제를 추가할 때는 하위 데이터 존재 여부를 서버에서 먼저 확인한다.
6. 종별 셀에는 참가자 수, 세부종목 수, 점수 수를 표시해 변경 영향을 사전에 이해할 수 있게 한다.
7. 비활성 종별은 관리자 직접 URL을 열어도 읽기만 허용하고, 선수 등록·세부종목·조 편성·레인 배정·점수 입력 write action은 서버에서 거부한다.
8. Supabase 공개 정책과 `participant_names` 뷰도 활성 종별과 그 하위 이벤트에서 파생되는 데이터만 읽도록 제한한다.

## 사용자 흐름

- 새 종별: 체크하면 카테고리를 만들고 활성 상태로 표시한다.
- 활성 종별 해제: 데이터가 있으면 영향 요약 확인창을 보여주고, 확인 시 `is_active=false`로 바꾼다. 데이터가 없는 경우에도 삭제하지 않고 동일한 상태 변경을 사용한다.
- 비활성 종별 복구: 기존 카테고리 ID를 사용해 `is_active=true`로 바꾼다.
- 공개 페이지: 활성 종별만 노출한다. 비활성 종별의 직접 URL은 공개 데이터 보호를 위해 찾을 수 없게 한다.

## 서버 안전장치

- 활성화/비활성화 요청은 서버에서 대회와 카테고리의 관계를 확인한다.
- 기존 카테고리 변경에는 카테고리 ID를 사용하고, 신규 생성에만 연령·성별을 사용한다. 서로 모순되는 입력은 거부한다.
- 일반 토글 경로에는 카테고리 삭제가 없으며, 향후 영구 삭제 액션은 하위 이벤트·선수·점수 등이 있으면 거부해야 한다.
- 비활성 카테고리를 대상으로 하는 모든 운영 write action은 공통 활성 상태 검사를 통과해야 한다.
- 카테고리·이벤트·선수·팀·조·라인업·레인·게임 상태·랭킹·점수와 `participant_names` 공개 경로는 `is_active=true` 관계를 확인한다.
- 모든 Supabase 쿼리는 기존 query builder와 Zod 입력 검증을 사용한다.

## 검증 기준

- 데이터가 있는 종별을 해제한 뒤 관련 행 수가 줄지 않는다.
- 해제된 종별은 공개 목록과 공개 상세 URL에서 제외된다.
- 관리자 대회 상세에서 다시 활성화하면 기존 카테고리 ID와 데이터가 유지된다.
- 신규 종별 추가와 기존 비활성 종별 복구가 중복 카테고리를 만들지 않는다.
- 서버 액션 직접 호출로 종별 사용 중지 시 삭제가 발생하지 않는다.
- 비활성 카테고리에서 선수 등록·종목 추가·조/레인/점수 저장이 거부된다.
- Supabase 공개 정책 직접 조회에서도 비활성 카테고리와 하위 이벤트가 노출되지 않는다.
- 기존 도메인 테스트와 빌드가 통과한다.
